# Requirements Document

## Introduction

The Membership Benefit Card (MBC) is a government cooperative program that uses NFC cards as "member identity cards full of benefits." There is no centralized database — all data (identity, balance, visit status, transaction history) lives on the NFC card itself. This is a web-based Progressive Web Application (PWA) built with React that reads/writes NFC card data directly, operating fully offline without any backend or external API dependency. The primary platform is web (not native mobile), with all NFC transactions designed to work without internet connectivity after initial installation.

The application is designed with an **extensible service type architecture** that supports multiple business scenarios beyond parking. The system accommodates duration-based services (parking, bike rental, notebook/smartphone rental, gym sessions), visit-based services (restaurant visits, VIP lounge access), and loyalty/discount services (membership discounts, loyalty point accumulation). Each service type defines its own pricing strategy, unit of measurement, and transaction behavior through a configurable engine.

The primary and default implementation is the **parking scenario** from the KDX document (Rp 2.000/hour, ceiling rounding), which serves as the reference implementation for the extensible architecture.

The application provides four role modes within a single app: The Station (admin registration, top-up & service configuration), The Gate (generic check-in for any configured service), The Terminal (generic check-out, fare/fee calculation & deduction), and The Scout (member's digital pocket book for viewing card contents).

## Glossary

- **MBC_App**: The single-page React application that operates in four switchable role modes to manage NFC card interactions
- **NFC_Card**: The physical NFC-enabled card that stores all member data including identity, balance, visit status, device binding identifier, and transaction history
- **The_Station**: The admin role mode used by cooperative officers for member card registration, balance top-up, and service type configuration
- **The_Gate**: The entry point role mode that records check-in timestamps and the active service type onto the NFC card
- **The_Terminal**: The exit point role mode that calculates fees based on the active service type's pricing strategy, deducts from balance, and releases visit status on the card
- **The_Scout**: The read-only role mode that allows members to view their card contents (balance, status, history) by tapping
- **Member**: A registered cooperative member who holds a valid NFC card with encoded identity and balance data
- **Balance**: The monetary credit stored on the NFC card, denominated in Indonesian Rupiah (IDR)
- **Check_In_Status**: A flag stored on the NFC card indicating the member is currently checked in to a service, with an associated entry timestamp, service type identifier, and the Device_ID of the check-in device
- **Transaction_Log**: A record entry stored on the NFC card containing the amount, timestamp, activity type, and service type identifier
- **Silent_Shield**: The data protection mechanism that encrypts/obfuscates sensitive card data so it cannot be read plainly by third-party NFC applications
- **Simulation_Mode**: A testing feature available in The Gate that allows manipulation of the check-in timestamp to a past time for fee calculation testing
- **Service_Type**: A configurable definition that describes a specific business scenario (e.g., parking, bike-rental, gym-session, restaurant-visit, vip-access) with its own pricing strategy and unit of measurement
- **Pricing_Strategy**: A configuration object associated with a Service_Type that defines the rate per unit, unit type (per-hour, per-visit, flat-fee), rounding strategy (ceiling, floor, nearest), and currency denomination
- **Activity_Type**: A string identifier for the type of transaction recorded in the Transaction_Log (e.g., top-up, parking-fee, bike-rental, gym-session, restaurant-visit, vip-access)
- **Service_Registry**: The in-app configuration store that holds all available Service_Type definitions and their associated Pricing_Strategy configurations
- **Fare**: The calculated charge for a service usage, determined by the active Service_Type's Pricing_Strategy (default: parking at Rp 2.000 per hour with ceiling rounding)
- **Device_ID**: A unique identifier generated by the MBC_App on first launch and persisted in local storage, used to bind a check-in session to a specific physical device
- **Atomic_Write**: A write operation to the NFC card that either completes all data changes (balance deduction, status clearing, log writing) as a single unit, or reverts all changes if any step fails — ensuring no partial state is written
- **Manual_Calculation_Mode**: A fallback interface in The Terminal that allows the operator to manually input check-in time and service type to calculate the fee without reading or writing to the NFC card, used when NFC hardware fails during check-out
- **NFC_Capability_Status**: The detected state of Web NFC support on the current device, one of: supported (API available and permission granted), unsupported (API not available in browser/platform), permission_pending (API available but permission not yet requested), or permission_denied (user explicitly denied NFC access)

## Requirements

### Requirement 1: Role Mode Switching

**User Story:** As a cooperative officer or member, I want to switch between application roles, so that I can perform different operations using a single installed application.

#### Acceptance Criteria

1. THE MBC_App SHALL provide a role selection interface with four options: The Station, The Gate, The Terminal, and The Scout
2. WHEN a role is selected, THE MBC_App SHALL switch the active interface to the corresponding role mode
3. THE MBC_App SHALL display a clear visual indicator of the currently active role mode at all times

### Requirement 2: NFC Card Reading

**User Story:** As a cooperative officer or member, I want the application to read NFC card data, so that I can access the information stored on the card.

#### Acceptance Criteria

1. WHEN an NFC card is tapped against the device, THE MBC_App SHALL read the card data within 3 seconds
2. IF the NFC hardware is unavailable or permission is denied, THEN THE MBC_App SHALL display an informative error message explaining the issue
3. IF the tapped card contains invalid or corrupted data, THEN THE MBC_App SHALL display an error message indicating the card is unreadable
4. THE MBC_App SHALL request NFC permission from the device before attempting any read or write operation

### Requirement 3: NFC Card Writing

**User Story:** As a cooperative officer, I want the application to write data to NFC cards, so that member information and balances can be stored directly on the card.

#### Acceptance Criteria

1. WHEN a write operation is initiated, THE MBC_App SHALL encode and write the data to the NFC card
2. IF a write operation fails, THEN THE MBC_App SHALL display an error message and retain the previous card state without any partial data written
3. WHEN a write operation completes successfully, THE MBC_App SHALL display a confirmation message to the operator
4. THE MBC_App SHALL verify written data by reading back the card contents after each write operation
5. THE MBC_App SHALL perform write operations as an Atomic_Write where all data changes succeed together or all changes are reverted to the pre-operation state
6. IF a write operation fails mid-operation (partial write detected during verification), THEN THE MBC_App SHALL rollback the card data to the state captured before the write was initiated
7. THE MBC_App SHALL NOT display a success confirmation until the post-write verification read confirms data integrity

### Requirement 4: Member Registration (The Station)

**User Story:** As a cooperative officer, I want to register new member cards, so that cooperative members receive their personalized NFC identity cards.

#### Acceptance Criteria

1. WHILE The_Station mode is active, THE MBC_App SHALL display a registration form requesting member identity information
2. WHEN a registration form is submitted with a blank NFC card tapped, THE MBC_App SHALL write the member identity data to the NFC card with an initial balance of Rp 0
3. IF the NFC card already contains member data during registration, THEN THE MBC_App SHALL display a warning that the card is already registered
4. WHEN registration completes successfully, THE MBC_App SHALL display a confirmation with the registered member details

### Requirement 5: Balance Top-Up (The Station)

**User Story:** As a cooperative officer, I want to add balance to member cards, so that members can use their cards for service payments.

#### Acceptance Criteria

1. WHILE The_Station mode is active, THE MBC_App SHALL provide a top-up interface with an amount input field
2. WHEN a top-up amount is submitted and a registered NFC card is tapped, THE MBC_App SHALL add the specified amount to the existing card balance
3. WHEN a top-up operation completes successfully, THE MBC_App SHALL display the previous balance, top-up amount, and new balance
4. THE MBC_App SHALL record the top-up as a Transaction_Log entry with activity type "top-up" on the NFC card
5. IF the tapped card is not a registered member card, THEN THE MBC_App SHALL display an error indicating the card is not recognized

### Requirement 6: Generic Check-In Recording (The Gate)

**User Story:** As a gate operator, I want to record member entry timestamps along with the active service type on their cards, so that service usage duration or visit can be tracked upon exit.

#### Acceptance Criteria

1. WHILE The_Gate mode is active, THE MBC_App SHALL display a check-in interface showing the currently selected Service_Type and ready to accept NFC card taps
2. WHEN a registered NFC card without an active Check_In_Status is tapped, THE MBC_App SHALL write the current timestamp as the check-in time, record the active Service_Type identifier, store the current Device_ID, and set the Check_In_Status flag to active as a single Atomic_Write
3. IF a registered NFC card with an active Check_In_Status is tapped, THEN THE MBC_App SHALL reject the operation and display a message indicating the member has already checked in (preventing double tap-in)
4. WHEN check-in completes successfully, THE MBC_App SHALL display a confirmation with the member name, recorded entry time, and the service type name
5. IF the tapped card is not a registered member card, THEN THE MBC_App SHALL display an error indicating the card is not recognized
6. WHILE The_Gate mode is active, THE MBC_App SHALL allow the operator to select which Service_Type to use for check-in from the configured Service_Registry
7. IF the NFC scan fails during the check-in write operation, THEN THE MBC_App SHALL NOT record the check-in as successful and SHALL display an error instructing the operator to re-tap the card
8. THE MBC_App SHALL disable further tap acceptance for the same card until the current check-in write operation completes or fails (preventing rapid double-tap registration)

### Requirement 7: Simulation Mode (The Gate)

**User Story:** As a tester or operator, I want to manipulate the check-in timestamp to a past time, so that I can test fee calculations without waiting real hours.

#### Acceptance Criteria

1. WHILE The_Gate mode is active, THE MBC_App SHALL provide a toggle to enable Simulation_Mode
2. WHILE Simulation_Mode is enabled, THE MBC_App SHALL display a date-time picker allowing the operator to set a custom check-in timestamp in the past
3. WHEN a check-in is performed with Simulation_Mode enabled, THE MBC_App SHALL write the custom timestamp instead of the current time to the NFC card
4. THE MBC_App SHALL visually distinguish Simulation_Mode from normal operation mode with a clear indicator

### Requirement 8: Generic Check-Out and Fee Calculation (The Terminal)

**User Story:** As a terminal operator, I want to process member exits by calculating the fee based on the service type's pricing strategy and deducting it from the balance, so that service charges are applied correctly.

#### Acceptance Criteria

1. WHILE The_Terminal mode is active, THE MBC_App SHALL display a check-out interface ready to accept NFC card taps
2. WHEN a registered NFC card with an active Check_In_Status is tapped, THE MBC_App SHALL read the stored Service_Type identifier, check-in timestamp, and Device_ID from the card
3. IF the Device_ID stored on the card does not match the current device's Device_ID, THEN THE MBC_App SHALL reject the check-out and display a message instructing the member to return to the original check-in device
4. THE MBC_App SHALL retrieve the Pricing_Strategy associated with the stored Service_Type from the Service_Registry
5. THE MBC_App SHALL calculate the fee using the retrieved Pricing_Strategy (rate per unit, unit type, and rounding strategy)
6. WHEN the calculated fee is less than or equal to the card balance, THE MBC_App SHALL deduct the fee from the balance, clear the Check_In_Status, clear the stored Device_ID, and record the transaction as a Transaction_Log entry with the corresponding Activity_Type as a single Atomic_Write
7. IF the calculated fee exceeds the card balance, THEN THE MBC_App SHALL display the shortage amount and instruct the member to top-up at The Station without deducting any balance
8. IF a registered NFC card without an active Check_In_Status is tapped, THEN THE MBC_App SHALL reject the operation and display a message indicating the member has not checked in (preventing double tap-out)
9. WHEN check-out completes successfully, THE MBC_App SHALL display the service type name, entry time, exit time, duration or visit count, fee amount, and remaining balance
10. IF the Service_Type stored on the card is not found in the current Service_Registry, THEN THE MBC_App SHALL display an error indicating the service type is not recognized and suggest reconfiguration at The Station
11. IF the NFC scan fails during the check-out write operation, THEN THE MBC_App SHALL NOT deduct the balance and SHALL display an error instructing the operator to re-tap the card
12. THE MBC_App SHALL disable further tap acceptance for the same card until the current check-out write operation completes or fails (preventing rapid double-tap deduction)

### Requirement 9: Card Information Display (The Scout)

**User Story:** As a cooperative member, I want to view my card information by tapping, so that I can check my balance, visit status, and transaction history instantly.

#### Acceptance Criteria

1. WHILE The_Scout mode is active, THE MBC_App SHALL display a tap prompt inviting the member to place their card
2. WHEN a registered NFC card is tapped, THE MBC_App SHALL read and display the member identity, current balance, active Check_In_Status with service type name (if any), and the last 5 transaction logs
3. THE MBC_App SHALL display transaction logs showing the amount, timestamp, activity type, and service type for each entry
4. IF the tapped card is not a registered member card, THEN THE MBC_App SHALL display an error indicating the card is not recognized

### Requirement 10: Transaction Log Management

**User Story:** As a system, I want to maintain a rolling history of the last 5 transactions on each card, so that members can review their recent activity.

#### Acceptance Criteria

1. WHEN a new transaction occurs (top-up or service fee deduction), THE MBC_App SHALL append a Transaction_Log entry containing the amount, timestamp, activity type, and service type identifier to the NFC card
2. WHILE the NFC card contains 5 Transaction_Log entries, THE MBC_App SHALL remove the oldest entry before writing a new one to maintain a maximum of 5 entries
3. THE MBC_App SHALL store each Transaction_Log entry with the following fields: amount in IDR, timestamp in ISO 8601 format, activity type as a string identifier, and service type identifier

### Requirement 11: Data Protection (Silent Shield)

**User Story:** As a cooperative administrator, I want sensitive card data to be protected, so that third-party NFC reader applications cannot read member identity and balance information in plain text.

#### Acceptance Criteria

1. THE MBC_App SHALL encrypt or obfuscate member identity data before writing to the NFC card
2. THE MBC_App SHALL encrypt or obfuscate balance data before writing to the NFC card
3. WHEN reading card data, THE MBC_App SHALL decrypt or de-obfuscate the protected fields before displaying them
4. FOR ALL valid card data objects, encrypting then decrypting SHALL produce the original data (round-trip property)

### Requirement 12: Configurable Pricing Engine

**User Story:** As a system, I want to calculate service fees using a configurable pricing strategy per service type, so that different business scenarios can define their own rate structures.

#### Acceptance Criteria

1. THE MBC_App SHALL calculate fees using the formula defined by the active Service_Type's Pricing_Strategy: fee = rounding_strategy(usage_units) × rate_per_unit
2. THE MBC_App SHALL support the following unit types in a Pricing_Strategy: per-hour (duration-based), per-visit (count-based), and flat-fee (fixed amount)
3. THE MBC_App SHALL support the following rounding strategies: ceiling (round up), floor (round down), and nearest (round to nearest integer)
4. WHEN the Pricing_Strategy unit type is per-hour, THE MBC_App SHALL calculate usage_units as the elapsed time between check-in and check-out in hours
5. WHEN the Pricing_Strategy unit type is per-visit, THE MBC_App SHALL calculate the fee as a single rate_per_unit charge per check-in/check-out cycle
6. WHEN the Pricing_Strategy unit type is flat-fee, THE MBC_App SHALL charge the fixed rate_per_unit amount regardless of duration
7. THE MBC_App SHALL include a default Pricing_Strategy for parking: rate_per_unit = Rp 2.000, unit_type = per-hour, rounding_strategy = ceiling
8. THE MBC_App SHALL display the calculated fee breakdown showing the service type, usage measurement, rate applied, and total amount before deduction

### Requirement 13: NFC Card Data Schema Serialization

**User Story:** As a developer, I want a well-defined data schema for NFC card contents, so that data can be reliably written to and read from the limited card memory.

#### Acceptance Criteria

1. THE MBC_App SHALL define a card data schema containing: member identity fields, balance, check-in status flag, check-in timestamp, active service type identifier, bound Device_ID, and an array of up to 5 transaction log entries
2. THE MBC_App SHALL serialize the card data schema into a compact binary or encoded string format suitable for NFC card memory constraints
3. THE MBC_App SHALL deserialize NFC card raw data back into the structured card data schema
4. FOR ALL valid card data objects, serializing then deserializing SHALL produce an equivalent object (round-trip property)
5. IF deserialization encounters malformed data, THEN THE MBC_App SHALL return a descriptive error rather than corrupted data

### Requirement 14: Offline-First Web Application

**User Story:** As a cooperative officer, I want the application to be a web-based PWA that functions without internet connectivity, so that all NFC transactions operate offline after initial installation.

#### Acceptance Criteria

1. THE MBC_App SHALL be a Progressive Web Application (PWA) built on web technologies (React) that can be installed on devices via the browser
2. THE MBC_App SHALL perform all operations (registration, top-up, check-in, check-out, card reading) without requiring network connectivity
3. THE MBC_App SHALL load and function from cached application assets after the initial installation using a Service Worker
4. THE MBC_App SHALL not make any external API calls during normal card operations
5. THE MBC_App SHALL provide a web app manifest enabling add-to-home-screen installation on supported devices
6. WHEN the device is offline after initial installation, THE MBC_App SHALL serve all application assets from the Service Worker cache

### Requirement 15: Service Type Configuration (The Station)

**User Story:** As a cooperative officer, I want to configure which service types are available and their pricing rules, so that the system can be adapted to different business scenarios without code changes.

#### Acceptance Criteria

1. WHILE The_Station mode is active, THE MBC_App SHALL provide a service configuration interface for managing the Service_Registry
2. THE MBC_App SHALL allow adding a new Service_Type with the following fields: unique identifier, display name, Activity_Type label, and a Pricing_Strategy (rate_per_unit, unit_type, rounding_strategy)
3. THE MBC_App SHALL allow editing an existing Service_Type's display name and Pricing_Strategy
4. THE MBC_App SHALL allow removing a Service_Type from the Service_Registry
5. THE MBC_App SHALL persist the Service_Registry configuration in the device's local storage so it survives application restarts
6. THE MBC_App SHALL include a pre-configured default Service_Type for parking with identifier "parking", display name "Parkir", activity type "parking-fee", rate Rp 2.000, unit type per-hour, and rounding strategy ceiling
7. IF no Service_Registry configuration exists on first launch, THEN THE MBC_App SHALL initialize the Service_Registry with the default parking Service_Type

### Requirement 16: Extensible Service Type System

**User Story:** As a cooperative administrator, I want the system to support multiple business scenarios through configurable service types, so that the same card and application can be used for parking, bike rental, gym sessions, restaurant visits, VIP access, and other check-in/check-out based services.

#### Acceptance Criteria

1. THE MBC_App SHALL support defining Service_Types for duration-based services including but not limited to: parking, bike rental, notebook rental, smartphone rental, and gym sessions
2. THE MBC_App SHALL support defining Service_Types for visit-based services including but not limited to: restaurant visits, VIP lounge access, and facility entry
3. THE MBC_App SHALL support defining Service_Types for loyalty or discount services where the check-in/check-out cycle records a visit without fee deduction
4. THE MBC_App SHALL store the Service_Type identifier on the NFC card during check-in so The Terminal can determine the correct Pricing_Strategy at check-out
5. THE MBC_App SHALL allow multiple Service_Types to coexist in the Service_Registry simultaneously
6. WHEN a Service_Type with unit_type per-visit is used, THE MBC_App SHALL record the visit in the Transaction_Log without requiring duration calculation

### Requirement 17: Service Type Selection at The Gate

**User Story:** As a gate operator, I want to select which service type applies to the current check-in location, so that the correct pricing and tracking rules are applied when members tap their cards.

#### Acceptance Criteria

1. WHILE The_Gate mode is active, THE MBC_App SHALL display a service type selector populated from the Service_Registry
2. WHEN only one Service_Type is configured in the Service_Registry, THE MBC_App SHALL auto-select that service type without requiring operator interaction
3. WHEN multiple Service_Types are configured, THE MBC_App SHALL require the operator to select a service type before accepting card taps
4. THE MBC_App SHALL persist the last selected Service_Type in The Gate mode so the operator does not need to re-select on each check-in
5. IF the Service_Registry is empty, THEN THE MBC_App SHALL display a message instructing the operator to configure service types at The Station

### Requirement 18: Atomic Transaction Integrity

**User Story:** As a cooperative administrator, I want all NFC write operations to be atomic and protected against scan failures, so that no double deductions, partial writes, or phantom transactions occur.

#### Acceptance Criteria

1. THE MBC_App SHALL capture a snapshot of the card's current state (balance, Check_In_Status, Transaction_Log) before initiating any write operation
2. THE MBC_App SHALL treat the combination of fee deduction, Check_In_Status clearing, and Transaction_Log writing as a single Atomic_Write that either fully succeeds or fully reverts
3. IF the NFC connection is lost during a write operation, THEN THE MBC_App SHALL treat the operation as failed and NOT display a success confirmation
4. WHEN a write operation fails, THE MBC_App SHALL attempt to restore the card to the pre-operation snapshot state on the next successful NFC connection
5. THE MBC_App SHALL maintain a write-lock state during any active write operation, preventing a second tap from initiating a concurrent write to the same card
6. IF the post-write verification read detects data inconsistency with the intended write, THEN THE MBC_App SHALL flag the operation as failed and attempt rollback
7. THE MBC_App SHALL ensure that for any single check-in/check-out cycle, the balance is deducted exactly once regardless of how many times the card is tapped

### Requirement 19: Device Binding (Single Device Lock)

**User Story:** As a cooperative administrator, I want each check-in session to be bound to a specific device, so that check-out can only be processed on the same device that performed the check-in.

#### Acceptance Criteria

1. THE MBC_App SHALL generate a unique Device_ID on first launch and persist the Device_ID in the device's local storage
2. WHEN a check-in operation is performed at The Gate, THE MBC_App SHALL write the current device's Device_ID to the NFC card as part of the Check_In_Status data
3. WHEN a check-out operation is initiated at The Terminal, THE MBC_App SHALL compare the Device_ID stored on the card with the current device's Device_ID
4. IF the Device_ID on the card does not match the current device's Device_ID during check-out, THEN THE MBC_App SHALL reject the check-out and display a message identifying that the member must return to the original check-in device
5. THE MBC_App SHALL clear the stored Device_ID from the card when a check-out operation completes successfully
6. THE MBC_App SHALL persist the Device_ID in localStorage so the identifier survives application restarts
7. IF the Device_ID is missing from localStorage on application launch, THEN THE MBC_App SHALL generate a new Device_ID and display a warning that previous check-in sessions bound to the old Device_ID cannot be checked out on this device

### Requirement 20: Device Data Persistence and Error Handling

**User Story:** As a cooperative administrator, I want the device's local data storage to handle errors gracefully, so that the application remains functional even when storage is unavailable or full.

#### Acceptance Criteria

1. THE MBC_App SHALL persist the Service_Registry, Device_ID, and application state using localStorage as the storage mechanism
2. WHEN the MBC_App launches, THE MBC_App SHALL check whether localStorage is available and accessible on the device
3. IF localStorage is unavailable (e.g. private browsing mode, browser restriction), THEN THE MBC_App SHALL display an informative message explaining that the browser does not support local storage and the application cannot persist data between sessions
4. IF a localStorage write operation fails due to quota exceeded, THEN THE MBC_App SHALL display a warning to the operator indicating storage capacity is full and suggest clearing browser data
5. THE MBC_App SHALL validate the integrity of the Service_Registry data on each application launch by checking for required fields and valid structure using Zod schema validation
6. IF the Service_Registry data is missing or corrupted in localStorage, THEN THE MBC_App SHALL re-initialize with the default parking Service_Type configuration
7. THE MBC_App SHALL display a prominent warning when critical data (Device_ID or Service_Registry) has been reset due to storage loss or corruption, informing the operator of potential impact on active check-in sessions
8. THE MBC_App SHALL handle all storage errors gracefully without crashing, ensuring the application remains operational and business errors are communicated clearly to the user

### Requirement 21: Manual Fee Calculation Fallback

**User Story:** As a terminal operator, I want a manual fee calculation mode available when the NFC card cannot be read during check-out, so that I can determine the correct fee and collect payment outside the application.

#### Acceptance Criteria

1. WHILE The_Terminal mode is active, THE MBC_App SHALL provide a "Manual Calculation" button to enter Manual_Calculation_Mode
2. WHILE Manual_Calculation_Mode is active, THE MBC_App SHALL display input fields for: check-in timestamp (date and time picker) and Service_Type selection from the Service_Registry
3. WHEN the operator submits the manual calculation form, THE MBC_App SHALL calculate the fee using the selected Service_Type's Pricing_Strategy and the time difference between the entered check-in time and the current time
4. THE MBC_App SHALL display the calculated fee amount, duration, and pricing breakdown to the operator
5. THE MBC_App SHALL NOT write any data to the NFC card during Manual_Calculation_Mode operations
6. THE MBC_App SHALL clearly label the Manual_Calculation_Mode interface as "Manual / Offline Calculation" to distinguish it from normal NFC-based check-out
7. THE MBC_App SHALL NOT record any Transaction_Log entry on the card for manual calculations since no card write occurs
8. WHILE Manual_Calculation_Mode is active, THE MBC_App SHALL still accept NFC card taps for normal check-out processing if the operator switches back to normal mode

### Requirement 22: NFC Capability Detection and Graceful Degradation

**User Story:** As a user opening the application for the first time, I want to immediately know whether my device and browser support NFC, so that I don't waste time trying to use features that won't work on my device.

#### Acceptance Criteria

1. WHEN the MBC_App is launched, THE MBC_App SHALL check whether the Web NFC API (`NDEFReader`) is available in the current browser environment
2. IF the Web NFC API is not available (unsupported browser or platform), THEN THE MBC_App SHALL display a full-screen compatibility notice explaining that the device or browser does not support NFC and listing the minimum requirements (Chrome Android 89+)
3. IF the Web NFC API is available but permission has not been granted, THEN THE MBC_App SHALL prompt the user to grant NFC permission before allowing access to The Station, The Gate, or The Terminal modes
4. IF the user denies NFC permission, THEN THE MBC_App SHALL display a clear message explaining that NFC permission is required for card operations and provide instructions to re-enable it via browser settings
5. WHILE the Web NFC API is not available, THE MBC_App SHALL disable access to The Station, The Gate, and The Terminal modes and visually indicate they are unavailable
6. WHILE the Web NFC API is not available, THE MBC_App SHALL still allow access to The Scout mode in a limited read-only demonstration mode showing sample card data for evaluation purposes
7. THE MBC_App SHALL display the current NFC support status (supported, unsupported, permission pending, permission denied) in the role selection interface
8. THE MBC_App SHALL provide a "Check NFC Support" diagnostic button that re-evaluates NFC availability and permission status on demand
