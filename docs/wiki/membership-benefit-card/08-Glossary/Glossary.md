# Glossary

> Source: requirements.md — Glossary section

## Domain Terms

### MBC_App
The single-page React application that operates in four switchable role modes to manage NFC card interactions.

### NFC_Card
The physical NFC-enabled card that stores all member data including identity, balance, visit status, device binding identifier, and transaction history. See [Card Data Schema](../02-Data-Models/Card-Data-Schema).

### The_Station
The admin role mode used by cooperative officers for member card registration, balance top-up, and benefit type configuration. See [Station Interface](../05-UI-Components/Station-Interface).

### The_Gate
The entry point role mode that records check-in timestamps and the active benefit type onto the NFC card. See [Gate Interface](../05-UI-Components/Gate-Interface).

### The_Terminal
The exit point role mode that calculates fees based on the active benefit type's pricing strategy, deducts from balance, and releases visit status on the card. See [Terminal Interface](../05-UI-Components/Terminal-Interface).

### The_Scout
The read-only role mode that allows members to view their card contents (balance, status, history) by tapping. See [Scout Interface](../05-UI-Components/Scout-Interface).

### Member
A registered cooperative member who holds a valid NFC card with encoded identity and balance data.

### Balance
The monetary credit stored on the NFC card, denominated in Indonesian Rupiah (IDR). Always a non-negative integer.

### Check_In_Status
A flag stored on the NFC card indicating the member is currently checked in to a service, with an associated entry timestamp, service type identifier, and the Device_ID of the check-in device. See [Card Data Schema](../02-Data-Models/Card-Data-Schema).

### Transaction_Log
A record entry stored on the NFC card containing the amount, timestamp, activity type, and service type identifier. Rolling max of 5 entries. See [Card Data Schema](../02-Data-Models/Card-Data-Schema).

### Silent_Shield
The data protection mechanism that encrypts/obfuscates sensitive card data so it cannot be read plainly by third-party NFC applications. Uses AES-256-GCM. See [Silent Shield Encryption](../04-Technical-Flows/Silent-Shield-Encryption).

### Simulation_Mode
A testing feature available in The Gate that allows manipulation of the check-in timestamp to a past time for fee calculation testing. See [Check-In Flow](../03-Business-Flows/Check-In-Flow).

### Benefit_Type
A configurable definition that describes a specific business scenario (e.g., parking, bike-rental, gym-session) with its own pricing strategy and unit of measurement. See [Benefit Type Model](../02-Data-Models/Benefit-Type-Model).

### Pricing_Strategy
A configuration object associated with a Benefit_Type that defines the rate per unit, unit type (`per-hour`, `per-visit`, `flat-fee`), and rounding strategy (`ceiling`, `floor`, `nearest`). See [Pricing Engine](../04-Technical-Flows/Pricing-Engine).

### Activity_Type
A string identifier for the type of transaction recorded in the Transaction_Log (e.g., `top-up`, `parking-fee`, `bike-rental`).

### Benefit_Registry
The in-app configuration store that holds all available Benefit_Type definitions and their associated Pricing_Strategy configurations. Persisted via [Storage Architecture](../04-Technical-Flows/Storage-Architecture).

### Fare
The calculated charge for a service usage, determined by the active Benefit_Type's Pricing_Strategy. Default: parking at Rp 2.000 per hour with ceiling rounding.

### Device_ID
A unique identifier generated by the MBC_App on first launch and persisted in local storage, used to bind a check-in session to a specific physical device. See [Device Binding](../04-Technical-Flows/Device-Binding).

### Atomic_Write
A write operation to the NFC card that either completes all data changes as a single unit, or reverts all changes if any step fails. See [Atomic Write Pipeline](../04-Technical-Flows/Atomic-Write-Pipeline).

### Manual_Calculation_Mode
A fallback interface in The Terminal that allows the operator to manually input check-in time and service type to calculate the fee without reading or writing to the NFC card. See [Manual Calculation](../03-Business-Flows/Manual-Calculation).

### NFC_Capability_Status
The detected state of Web NFC support on the current device: `supported`, `unsupported`, `permission_pending`, or `permission_denied`. See [NFC Capability Detection](../04-Technical-Flows/NFC-Capability-Detection).

## Technical Terms

### Protocol
An interface defined in `@core/protocols/`. Protocols are framework-agnostic contracts that infrastructure adapters implement. Examples: `NfcProtocol`, `KeyValueStoreProtocol`.

### Service
Business logic in `@core/services/`. Can be pure (stateless) or stateful. Examples: `pricing.service`, `nfc.service`.

### Use Case
An orchestrator in `@core/use_case/` with single responsibility. Each use case is a single function: `execute(input) → result`. Examples: `CheckIn`, `CheckOut`.

### Controller
A pure factory function in `controllers/` that receives all dependencies via Awilix DI and returns a typed interface for the view. Uses only `import type`.

### Awilix
The dependency injection container used throughout the project. All runtime dependencies flow through `AwilixRegistry`.

### AwilixRegistry
The TypeScript type union of all DI container interfaces. Provides full type safety across the DI boundary.

## Currency

- **IDR** — Indonesian Rupiah
- **Rp** — Rupiah symbol used in display (e.g., "Rp 2.000")
- All monetary values are stored as integers (no decimals)
- Formatted using `formatIDR()` helper
